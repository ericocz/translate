# 沉浸式翻译 — 服务端化 + 账号体系 + 管理台 产品设计文档

> 本文是「把翻译能力服务端化、引入账号与运营能力、准备上架 Chrome/Edge 市场」这次重大重构的产品设计基准。
> 与 [`CLAUDE.md`](CLAUDE.md)（开发基准）、[`沉浸式翻译插件-用户体验设计.md`](沉浸式翻译插件-用户体验设计.md)（体验设计）配套。
> 状态：方案已确认，按文末「分阶段路线图」逐阶段各自走实现计划。

## Context（为什么做这次重构）

当前插件是**纯客户端、自用**形态：DeepSeek API Key 随构建注入进扩展包、翻译缓存在每台设备的 IndexedDB 里各存各的、提示词与流水线全在扩展内（`lib/config.ts` / `lib/deepseek.ts` / `lib/cache.ts` / `lib/translator.ts` / `lib/prompt.ts`）。

要发布到 **Chrome / Edge 应用市场**面向公众，这套形态有三个硬伤：

1. **API Key 暴露**——key 在产物 JS 里，任何人解包即可盗用，无法对外发布。
2. **缓存不共享**——每个用户重复翻同一篇公开文章，Token 成本随用户数线性增长，烧不起。
3. **无法运营**——没有账号、用量、限流、打点、错误上报，既挡不住刷量，也看不到线上发生了什么。

**本次重构的本质**：把 `translator + deepseek + cache + prompt + config` 这条**翻译流水线整体搬到 FastAPI 后端**，扩展退化为「DOM 端 + 后端 API 客户端」。配套建账号体系、梯度限流、打点/错误上报，以及一个 Next.js 管理台。

**预期结果**：可安全公开发布的扩展；上游 Key 只存服务端；全局共享缓存把 Token 成本从「随用户数线性增长」压到「随**新内容**增长」；登录用户免费畅用、匿名用户每天 3 页、有完整的防刷与可观测能力。

## 目标与非目标

**目标**
- 上游 DeepSeek Key 永不出服务端；扩展产物里不再有任何密钥。
- 翻译流水线全部行为**逐条保住**：稳定前缀缓存、`thinking:disabled`、分批（~40 块）、按 source 去重、`[[id]]` 流式切块（正则含 `.`）、标记平衡校验、原样回显不入缓存。
- 全局共享缓存；缓存命中也能给用户**记 Token**（输入+输出）。
- 匿名 3 页/天闸门 + 邮箱密码账号；登录后免页数限制。
- 梯度限流（按日 Token 上限分档，升降档都提醒用户）。
- 打点与错误上报落库，管理台可查日志、看统计、管用户、管上游 Key。

**非目标（本期不做）**
- 付费订阅 / 计费（仅预留数据结构，不实现支付）。
- 常驻 MutationObserver / 动态界面深度适配（维持现有「加载时内容 + 有界沉降补抽 + SPA 软导航」边界）。
- 多语言互译（仍只做英→中）。
- 客户端不再保留翻译缓存（缓存改为服务端唯一；客户端仅保留 Ctrl+点击还原所需的「原文 HTML」缓冲，与翻译缓存无关）。

## 目标架构

```
┌─────────────────────────────┐        ┌──────────────────────────────────────┐
│  扩展（Chrome / Edge MV3）   │        │            FastAPI 后端                │
│                             │  HTTPS │                                        │
│ content.ts                  │  (SSE) │  /v1/translate   翻译编排 + 流式回送   │
│  抽取 / <gN>标记 / styleMap  │ ─────▶ │   缓存优先 → 去重 → 分批 → DeepSeek    │
│  重建 DOM / Ctrl+点击还原    │ ◀───── │   → 标记校验 → 写缓存 → Token 记账     │

│ background.ts               │        │  /v1/usage       配额 / 当日用量        │
│  port 适配 → 调后端 API      │        │  /v1/events      打点上报              │
│  图标三态 / SPA 监听         │        │  /v1/errors      错误上报              │
│                             │        │                  ▲                     │
│ lib/api.ts（新）             │        │  /admin/* (内部) │  Next.js 管理台 ───┘
│  鉴权头 / device id / SSE    │        │                                        │
│ popup：登录入口 + 配额提示    │        │           PostgreSQL                    │
└─────────────────────────────┘        │  users / sessions / translation_cache  │
                                       │  daily_usage / quota_tier / events     │
        ┌──────────────────┐           │  error_logs / upstream_keys / admins   │
        │  DeepSeek V4Flash │◀──────────┤  （DeepSeek Key 仅存这里的 env/表）     │

**职责切分原则**：客户端只碰 DOM（抽取、标记、styleMap、重建、还原、流式回填的视觉过渡）；后端拥有「翻译这件事」的全部业务（密钥、提示词、缓存、模型调用、校验、记账、限流）。styleMap 仍**只在客户端**，发给后端的是带 `<gN>` 标记的 source 行，后端再转发给模型——与现有铁律 3 一致。

## 后端（FastAPI + PostgreSQL）

**分层**：`routers/`（HTTP+SSE）· `services/`（翻译编排、限流、记账、打点）· `repositories/`（Postgres 访问）· `core/`（config、JWT、密码哈希）。翻译编排 service 是把现有 `lib/translator.ts` + `lib/deepseek.ts` 用 Python 重写——逻辑等价、**注释里标注对应的客户端铁律**。

### 数据模型（Postgres 主要表）

| 表 | 关键列 | 说明 |
|---|---|---|
| `users` | id, email(uniq), password_hash, tz, created_at | 邮箱+密码账号 |
| `sessions` | id, user_id, refresh_token_hash, expires_at | 刷新令牌（access 用短时 JWT，无需存） |
| `translation_cache` | key(pk)=`{version}:{cyrb53(source)}`, translated, **input_tokens**, **output_tokens**, hits, last_access | 全局共享缓存；token 列是缓存命中记账的依据 |
| `anon_usage` | device_id, local_date, page_key, created_at（uniq: device+date+page_key） | 匿名「每页一次」去重计数；`page_key`=规范化 URL 哈希 |
| `daily_usage` | user_id, local_date, input_tokens, output_tokens, pages（pk: user+date） | 登录用户当日累计用量（含缓存命中归因） |
| `quota_tier` | user_id(pk), tier, daily_cap, strikes, clean_days, updated_at | 梯度限流状态机：当前档位/日上限/违规计数/连续达标天数 |
| `events` | id, ts, user_id?, device_id?, type, host, props(jsonb) | 运营打点（host 而非全 URL/正文） |
| `error_logs` | id, ts, user_id?, device_id?, kind, message, context(jsonb) | 客户端/服务端错误上报 |
| `upstream_keys` | id, label, key_ciphertext, status, used_tokens, balance_note | 上游 DeepSeek Key 池（加密存储，可轮换/停用） |
| `admins` | id, email, password_hash, role | 管理台登录（与终端用户隔离） |

### API 表面

| 端点 | 用途 |
|---|---|
| `POST /v1/translate` | **核心**。Body: `{blocks:[{id,source}], localDate}` + 鉴权头/device id。SSE 流：`block`/`done`/`error`/`notice`/`quota` 事件 |
| `POST /v1/auth/register` · `/login` · `/refresh` · `/logout` | 邮箱密码、JWT 颁发与刷新 |
| `GET /v1/usage` | 当前账号/设备的配额与当日用量（popup 顶部用） |
| `POST /v1/events` · `POST /v1/errors` | 打点与错误上报（批量、fire-and-forget） |
| `/admin/*` | 管理台后端（管理员 JWT 鉴权）：用户、用量统计、日志、上游 Key CRUD |

## 关键流程

### 1) 流式翻译（缓存 + 记账，替代现客户端流水线）

1. 客户端 `content.ts` 抽取出带 `<gN>` 标记的块（styleMap 留本地），`background.ts` 经 `lib/api.ts` `POST /v1/translate`。
2. 后端鉴权/识别身份（登录用户 or device id）→ **先查配额**（见流程 2/3），超限直接发 `quota` 事件并结束。
3. **缓存优先**：`translation_cache` 批量查 → 命中块立即随 SSE `block` 事件回送，并把该块的 `input_tokens+output_tokens` 计入用户当日用量（**缓存命中也记账**）。
4. 未命中块：按 source 去重 → 分批（~40）+ 有限并发 → 带稳定 `SYSTEM_PROMPT` 前缀、`thinking:disabled` 调 DeepSeek（SSE）。
5. 后端在**完整缓冲**上按 `[[id]]`（正则含 `.`）切块 → 标记平衡校验 → 通过则 SSE `block` 回送、写缓存、记 token；不缓存原样回显。
6. 客户端收 `block`：本地再校验一次标记 → 用 styleMap 重建 DOM → 淡入替换。`done` 收尾，`error` 按类提示。
7. 全程稳定前缀让**所有用户共享同一份 DeepSeek 前缀缓存**，命中率高于单机时代。

**客户端↔后端流协议**：后端直接发**结构化 SSE 事件**（`event: block` + `data: {"id":"b1","translated":"…"}`），客户端无需再做模型级 token 切块——切块在服务端完成。`notice` 事件承载限流提醒文案，`quota` 事件承载「需登录/额度满」。

### 2) 匿名配额（每页一次，3 页/天）

- 客户端带 `deviceId` + `localDate`（用户时区）+ 当前页 `page_key`（规范化 URL 哈希）。
- 后端：若 `(device, local_date)` 下**不同 `page_key` 数 ≥ 3** 且本页是新页 → 发 `quota`（「今日 3 页已用完，登录后免费畅用」）。
- 同一 `page_key` 重复翻译（刷新/重进）**不占新额度**——满足「每页一次」。
- IP 作为软兜底（防同设备清空 storage 刷量），仅记录不硬卡。

### 3) Token 记账 + 缓存命中归因（已简化）

两条独立、各自简单的路径，**不做批次→单块的比例分摊**：

- **未命中（调模型）**：直接以 DeepSeek 接口返回的 `usage`（`prompt_tokens + completion_tokens`）为准，按批次把**真实用量**累加进用户当日 `daily_usage`。这是权威口径。
- **写缓存**：每个块翻译后，用轻量本地 token 估算器（按字符 / CJK 近似）算出该块**自身**的输入/输出 token，写入 `translation_cache.input_tokens/output_tokens`。无需触碰批次总量、无分摊。
- **命中（缓存）**：直接读出缓存里记下的该块输入/输出 token，累加进用户当日用量——**缓存命中也记账**，反映「交付给用户的价值」，与限流口径一致。
- **部分变化的页**：每个块各走各路（命中读缓存 token、未命中用 API `usage`），不为「页面整体变了一点」做任何特殊对账——即满足「即使有些段落变化」。
- 不重复计：同一次请求内一个块只走其中一条路；跨次访问各自计费（每次访问都消耗当日用量），符合预期。

### 4) 梯度限流（仅以固定日 Token 上限为信号）

`quota_tier` 是个简单状态机，每用户一行：
- **档位与上限**：`tier 0`=慷慨上限 `CAP0`（够重度正常阅读）；`tier 1`=`CAP1<CAP0`；`tier 2`=`CAP2<CAP1`…
- **触发降档**：某本地日 `daily_usage` 达到当前 `daily_cap` → 记一次 `strike` + 当次请求发 `notice`「今日额度已达上限」。`strike` 在窗口内累计到阈值（如连续多日顶格）→ 降一档（`daily_cap` 下调）+ 发 `notice`「检测到异常用量，额度已临时下调」。仍顶格 → 再降一档。
- **逐步恢复**：连续 `clean_days`（当日用量 < 上限某比例）累计到阈值 → 升一档、`daily_cap` 回升 + 发 `notice`「用量恢复正常，额度已回升」。
- **每次升/降档都经 `notice` 事件 → popup 提示用户**（满足要求 (d)）。
- 普通用户几乎永不触档（`CAP0` 设在远超人工阅读的水位），只有脚本式刷量者会被逐级收紧。

### 5) 打点与错误上报

- 客户端在关键节点埋点（翻译开始/完成/失败、块数、缓存命中率、首块延迟、批次 token），批量 `POST /v1/events`，fire-and-forget 不阻塞翻译。
- 失败走 `POST /v1/errors`（含失败分类 + 脱敏上下文）。
- **隐私**：只上报 `host`（不含完整 URL、不含页面正文/译文）；缓存里存的是公开网页段落文本——隐私政策需明示。API Key 绝不出现在任何事件/日志（延续铁律 9）。

## 管理台（Next.js）

**技术选型（推荐）**：**Next.js App Router + shadcn/ui + Tremor（图表）+ TanStack Table/Query**，以 REST 直连 FastAPI `/admin/*`。理由：稳定、资料极丰富、与本项目「轻量可组合、不上重框架」的偏好一致，CRUD+图表+表格刚好够用，无额外框架魔法。
- 备选：**Refine**（refine.dev，更「开箱即用」的 admin 框架，自带 data/auth provider），若希望少写 CRUD 样板可选，但引入一层抽象。

**功能模块**
1. **日志查看**：`error_logs` 检索/筛选（按时间/用户/类型/host）。
2. **打点统计**：`events` 聚合看板——DAU、翻译页数、缓存命中率、Token 消耗趋势、Top 域名。
3. **用户管理**：用户列表、当日/历史用量、当前限流档位，可手动调档/封禁。
4. **API Key 管理**：上游 DeepSeek Key 池——增删、轮换、停用、查看各 Key 用量/余额备注。
- **统计核心**：**每用户每日 Token**（输入/输出）来自 `daily_usage`，**含缓存命中归因**（见流程 3）。

## 技术与安全

- **后端**：FastAPI（async）+ Postgres（SQLAlchemy/asyncpg + Alembic 迁移）；SSE 用 `StreamingResponse`。
- **鉴权**：终端用户短时 access JWT + 长效 refresh（`sessions` 存哈希）；密码 `argon2`/`bcrypt`；管理员独立 `admins` 表与登录。
- **密钥**：DeepSeek Key 存服务端 env（单 Key）或 `upstream_keys` 加密列（多 Key 轮换）；**永不下发客户端、永不入日志/事件**。
- **部署**：容器化 FastAPI + 托管 Postgres（Fly.io / Railway / Render 或 VPS）；后端需能直连 `api.deepseek.com`。**[开放项]** 具体托管与地域待定（影响全球用户与对华可达性）。
- **发布合规**：因新增账号与数据采集，上架前需备**隐私政策页**、权限用途说明、用户协议（Chrome/Edge 商店要求）。
