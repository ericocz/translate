### 🔧 方法论修正：ext=0 多为驱动假阴性，非 bug
驱动批次里 4 个 ext=0 经 MCP 复核**全部正常翻译**，原因分类：
- **重定向**：`docs.cursor.com`→`cursor.com`（白名单需跟到最终域名）。
- **重型 SPA / 超大页**：Stripe quickstart、SO 问题页（2144块）、CSS-Tricks flexbox（2140块）流式翻译慢，驱动 12s 早退误判 → 已把驱动早退阈值放宽到 22s + 块级元素<8 才早退，`MAX` 提到 48s。
- 教训：**ext=0 一律要 MCP 复核**，区分"真没翻"与"还没翻完/域名没命中"。

### 🔧 方法论修正 2：驱动加固（runner2.mjs）
旧驱动（runner.mjs）有两个问题，导致 batch-4/5 大量假 ext=0 且中途卡死：
- **导航后固定 sleep(3500) 不够**：代理 + 快速连续导航下页面常没加载完就查 → 假 ext=0（PyPI/SourceForge/SuperUser/Bitbucket 经加固驱动复核全部正常翻译）。
- **标签页崩溃/关闭无恢复**：重型站（Ars Technica #25、Kubernetes #72 附近）触发 "Inspected target navigated or closed"，旧驱动整个卡死，后续站全未测（batch-4 停在 #25、batch-5 停在 #72）。
- **加固版 runner2.mjs**：① 导航后先等 `readyState==='complete'` 再查；② ext=0 自动多等一轮复查；③ 标签页崩溃自动重建标签页并重试（日志带 `*`）；④ CDP 调用 60s 超时防永久挂起。复跑确认假阴性消除、不再卡死。
- **站点自身 JS 报错 ≠ 扩展 bug**：如 Bitbucket 的 `Uncaught TypeError`、新闻站的各种报错，多为站点自有；只有 `removeChild/NotFoundError → 整页崩溃` 才是要防的（dom-compat 已兜住，全程未见）。

### ⚠️ Reddit（#29）— 主内容译出且稳定，shadow DOM 内容不翻（已知限制）
- 驱动报 ext=0 是**慢 SPA 假阴性**（`shreddit-async-loader` 流式注入内容，需 5s+）；MCP 复核：transIds=208、帖子标题/规则/侧栏/公告均译成中文。
- **延迟复查（滚动 + 等 12s）：transIds 208→208、中文块 26→26、`reverted:false`** → **#29 的"动态重渲染把中文打回英文"担忧不成立**，翻译稳定。
- **限制（架构）**：页面有 **1030 个 shadow root**（`faceplate-*`/`rpl-*`/`shreddit-*` web components），其中约 2957 字文字在 shadow DOM 里；`lib/extractor.ts` 用 `document.createTreeWalker(document.body)`，**TreeWalker 不进 shadow root** → 这部分（左侧 Home/Popular/News 导航、广告帖、部分计数标签）保持英文。主可读内容在光 DOM、正常翻译。
- 判定：**非 bug，是已知限制**（"仅处理光 DOM；shadow DOM / 异步追加内容不翻"）。要支持需扩展抽取器递归 `shadowRoot`（非平凡，影响面广），收益（Reddit 等少数 web-component 站的 UI chrome）有限，暂不做。

### 已知限制补充（本轮新确认）
- **shadow DOM 内容不翻**：见上 Reddit。同理适用于任何把内容放进 shadow root 的 web-component 站。
- **重型 SPA / 超大页加载慢**：内容流式注入，需更长等待才出全译文（Stripe quickstart、SO/CSS-Tricks 2000+ 块、Reddit、wikiHow Tie-a-Tie、freeCodeCamp 索引、Notion）。非 bug，但自动化测试要给足等待。
- **重定向 / 多域名**：docs.cursor.com→cursor.com、notion.so↔notion.com——按域名白名单时要跟到最终域名。

## 🟢 后续修复：晚渲染 SPA ext=0（#3-B）已实现并验证
