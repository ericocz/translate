三条经营原则（于东来式，写进产品而不是挂在嘴上）

① 高质量、不吹牛
宣称的功能维度与实际严格一致（需求十九）。上线 Gate（§15）里每一条都有验收标准，没过就不上、不提前宣传。
② 定价透明、口径透明
已拍板：透明做到「定价层」——价格表、每次翻译扣多少额度怎么算、充值手续费占比，全部在产品内可查。不公示经营账目（D-08）。
③ 永远让用户有退路
缓存可关、额度可查、注册码邮箱可自助重查、数据可删。盈利后追加释放免费名额（需求四 c）——承诺写进产品页。
01

第一性原理：把这门生意拆到物理底层

为什么这个模式烧不死

翻译产品的本质是一道恒等式：经营成本 = Token 单价 × 消耗量 + 固定开销。通用翻译市场价格地板是 0（巨头免费、开源自部署），正面拼「更便宜」没有出路。V2 的解法是把成本结构按用户类型拆开，逐类封顶：

用户类型谁出 Token 钱我们的边际成本敞口上限角色
买断用户（¥39.9/$7）用户自己的 API Key（BYOK）≈ 0 翻译流量不过我们服务器无敞口利润 + 口碑主力
充值用户（¥1–100/$1–20）预付额度，按量扣减负成本 定价 ≥ 成本 × 安全系数预付封顶，永不垫资现金流
免费领取者（¥2/$0.5）我们（获客投资）有限 单人封顶 ¥2/$0.55000 名额 ≈ ¥1.0–1.8 万总敞口获客漏斗顶部
未领取的游客—0（D-04：不保留每日免费）无敞口只能看，引导领取/付费
第一性结论：这个模式下不存在「用户越多亏越多」的死亡螺旋。最坏情况 = 5000 名额全领全用完 + 没人付费，总损失约一万多元人民币——这是一次性、封顶的获客预算，不是无底洞。而任何一个买断/充值用户都是正贡献。个人开发者最稀缺的是「不死」，这个结构保证不死。

为什么敢透明？

DeepSeek 的刊例价是公开的（§9），瞒也瞒不住。我们卖的从来不是「转售 Token 的差价」，而是工程价值：标记保真（<gN> 体系）、全局缓存命中省下的钱、字幕/聊天站/输入框这些场景化能力、视角稳定不跳动的体验。把成本口径摊开说，反而把竞争对手拖进「他们为什么不敢说」的尴尬里——透明本身就是武器。

与「论文精读」的关系（D-01）

2026-06-11 的判断依然成立：通用翻译获客难、分发贵。所以：通用翻译是底座（本蓝图全部内容），科研论文精读是尖刀（后续付费深水区）。先用本蓝图完成可收费的通用产品，再在 arXiv/学术场景上做公式保真、术语统一、AI 速读的差异化付费能力——获客落点仍然打学术人群。两者共用同一套额度/账号/收款地基，不是两个产品。

原需求有两处按原文实现会违规，已在蓝图中修正：
① 需求二「通过浏览器指纹识别用户」若默认无感采集——指纹属个人数据，且读取终端信息受 ePrivacy 约束，纯防薅理由主张「严格必要」很勉强。修正：只在用户点击「领取」按钮的同意流程后采集（按钮上方明示用途与存储形式），不领取就永不采集。仪式感按钮恰好就是合规的同意时刻——产品设计与法务要求在这里天然合一。

  • 自建 vs 开源：起步自建（表都有了，加一个 worker 而已）。若错误聚合/去重需求长大，再上 GlitchTip（Sentry 协议兼容、单容器可跑，个人 VPS 扛得住）——不上 Sentry 自部署（Kafka+ClickHouse 全家桶，个人维护是灾难）。

体验军规

需求十六 · 十八 · 十九 · 宣称即兑现

军规一：翻译过程视角零移动（需求十九）

  • 替换前记录视口锚点（视口内第一个可见块 + 相对偏移），整批替换在 requestAnimationFrame 内合帧完成后按锚点校正 scrollTop。
  • 利用浏览器原生 scroll anchoring（overflow-anchor），并保证我们不触发它的失效条件（不在视口上方做无占位的高度突变）。
  • 双语对照插入译文导致的高度增长：视口上方的块先占位后填充（min-height 预留），淡入只动透明度不动布局。
  • 验收标准（进 §15 Gate）：长文页翻译全程，视口内文字位移 ≤ 1 行高。

军规二：日文专项（日本网友的三连击，逐条还击）

问题根因修法
① 汉字形似词被当成中文直译划词只看单词本身，孤词语言检测对 CJK 汉字词必然失准划词请求强制携带页面声明语言 + 前后 ~50 字上下文；判定优先级：页面 lang 属性 > 上下文统计 > 词形。在 lang=ja 页面上，汉字词一律按日文词处理。
② 已识别为日文、标了假名，朗读仍念汉语拼音TTS 没按语言选 voice，拿了系统默认中文语音speechSynthesis 选 voice 必须 lang 前缀匹配 ja-*，且 utterance.lang='ja-JP'；找不到日语语音就禁用朗读按钮并提示安装语音包——绝不静默 fallback 到中文。其他语言同规则（通用修复，不止日文）。
③ 形近词整体体验—划词卡片明示检测到的源语言（可点击切换重译），用户一眼可纠错。日文用例进回归集（《测试网站清单》补日文站）。

军规三：输入框三空格（需求六.4 + ComfyUI 修复）

  • 监听改为 document 捕获阶段 + composedPath()，穿透 shadow DOM 与动态创建的输入元素——ComfyUI 的画布弹出 textarea、CodeMirror、contenteditable 一并覆盖（这正是 ComfyUI 失灵的根因：旧实现只扫初始 DOM）。
  • 首次触发：弹窗三选「开启 / 关闭 / 下次再说」（需求六原文）；「下次再说」= 本次会话不再提示，下次触发再问。
  • 翻译后原文压入本地栈，再按一次快捷键 / 点击输入框图标可回看与还原原文（需求六「返回输入框查看之前的原文」）。
  • 验收：ComfyUI 文本框三空格触发成功率 100%（进 Gate）。

军规四：聊天站实时翻译（需求十八）

  • 站点适配器声明「消息容器」选择器（Discord 首发，Telegram Web/Slack 次批）；MutationObserver 只挂容器，不挂全文档。
  • 新消息节点入队，500ms 合批翻译（省请求省额度）；WeakSet 标记已译节点防重；翻译插入用占位策略维持滚动位置（军规一同款，聊天流场景加「贴底跟随」豁免：用户本就贴底时允许自然下滑）。
  • 速率保险丝：单页每分钟翻译块数上限（默认 120 块/分，O-17 可调），刷屏频道不至于静默吃光额度；触顶时图标提示「已暂停自动翻译，点击恢复」。
