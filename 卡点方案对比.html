两大卡点 · 方案对比与建议

2026-06-14 · 已核对 Stripe / Chrome 当前政策 · 数字为开办区间估算，最终以各官网当下为准

卡点 1 海外收款主体（开 Stripe 的前提）高·需先决策

核心事实：Stripe 香港只接受公司主体（limited company），不收个人户；国内个人身份无法凭空开。 另一个常见误解先澄清——Payoneer 不是主体方案，它只是「把钱从 Stripe / 平台提到国内卡」的收款落地工具，三个方案里都用得上，但它替代不了主体。真正要在下面三条路里选一条：

方案一次性 + 年成本费率可行性 / 优势代价 / 风险
A. 香港公司 + Stripe HK
最贴合原蓝图		 注册 ¥3–5k
年续费/秘书+审计 ¥5–9k		 ~3.4% + HK$2.35 KYC 通过率最高、最稳，主体真实可长期经营、开对公账户、报税清晰。香港无外汇管制、无需备案。 开办与每年成本最高；需香港注册地址（秘书服务）、每年做账审计；要配香港虚拟银行账户（Airwallex/空中云汇 等）。
B. 美国 LLC（Stripe Atlas）+ Stripe US
 Atlas 一次性 ~$500
州年费 $60–300（如 WY/DE）		 ~2.9% + $0.3（更低） 非美居民可办；Atlas 一条龙给 LLC + EIN + Stripe US；配 Mercury/Wise 收款。费率比香港低。 需处理美国年报与合规报税（5472/1120 等，常需找代理）；银行开户偶有拒；与「香港部署」无关联但不冲突。
C. MoR 平台（Paddle / Lemon Squeezy / Polar）
零主体起步		 ¥0 开办 ~5% + 手续费 无需任何公司主体即可上线收款；平台是「记录商户」，代收代缴全球税(VAT/GST)，打款到你 Payoneer/银行。国内独立开发者最省事的起步路。 费率最高（吃掉 ~5%）；与原蓝图「自有 Stripe 买断」决策冲突——你此前刚去掉 Creem（Creem 正是一家 MoR），需想清当时弃用它的理由是否同样适用于 Paddle 等。
关键澄清：原清单里「Stripe + Payoneer」其实仍隐含需要一个主体（A 或 B）才能开 Stripe。若你想先零成本验证付费转化、再补主体，C 是唯一能立刻上线的路；跑通量级后再迁到 A/B 拿低费率。
建议：按你的目标分两步走—— ① 先用 C（MoR）上线，零主体、当天能收钱，验证「老外愿不愿为 $9.99 买断付费」； ② 月流水站稳后再上 B（美国 LLC，费率最低、办起来比香港轻）作为长期主通道，把 ~5% 降到 ~3%。 只有当你确定要长期重运营、需要香港对公与报税体系时，A 才值得它的高成本。

若坚持原蓝图「就用自有 Stripe」，则直接选 B（比 A 便宜、费率更低），跳过 C。

卡点 2 商店「外部支付」合规低·基本可放心

核对后可下调风险：Google 早在 2021-02 关停了自有的 Chrome Web Store Payments，此后官方明确开发者可自由使用任意第三方支付（Stripe/微信/支付宝等）。Edge 同样无内购、允许外部支付。买断/充值走你自己的收款通道是合规常规做法，不是灰色地带。只要守住下面几条即可：

合规红线怎么做才安全
透明披露定价 商店详情页 + 扩展内如实写清价格、币种、是买断还是充值、退款政策。不得有诱导/隐藏收费。
支付页放在网页、不在扩展内嵌 点击后跳转到你域名下的结账页完成支付，而非在扩展弹窗里嵌入支付表单——既合规又规避 MV3 限制。
MV3 禁止远程代码 扩展包内不得动态加载远程 JS。支付逻辑全部由网页承载，扩展只负责跳转 + 回调校验。你后端已是 SSE 架构，天然契合。
数据/隐私声明一致 隐私政策 URL（《隐私政策.md》挂网）需与实际数据行为一致；涉及账号/支付要在商店数据用途里勾选准确。
结论：这条不构成阻塞，无需线下办理，只是开发/上架时遵守的几条规范。原清单 #18 的 ⚠ 可降级为「普通检查项」。唯一要做对的动作是：支付走网页结账页 + 扩展只跳转。
