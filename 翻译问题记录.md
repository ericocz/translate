# 翻译问题记录（经验库）

本文件记录"某些页面翻译时出问题"的案例，供后续修复时参考。
核心目的有两个：**① 遇到新问题先来这里查有没有同类前例；② 修复后回填，并把每条案例的"复现与验证"当成回归清单——以后改了别处，回来逐条复测，防止老问题复发。**

> 给未来的 Claude / 维护者：
> - **动手修任何"页面翻译异常"前，先通读本文件**，特别是"问题原因"和"涉及模块"，很可能正在改的就是某条案例的成因附近。
> - **修完后必须做两件事**：把新案例按模板追加到下方；对"涉及模块"相同的历史案例，逐条跑一遍它的"复现与验证"，确认没被改回去。
> - 若某条老问题复发，不要新建案例，而是把原案例状态改为 `⚠️ 复发`，在"复发记录"里追加日期与原因。

## 案例

### #1 Docusaurus logo 图片被放大

- **发现日期**：2026-06-05
- **状态**：✅ 已修复（2026-06-05）
- **URL**：https://hermes-agent.nousresearch.com/docs
- **站点 / 框架**：hermes-agent.nousresearch.com，**Docusaurus**（`div#__docusaurus`、`navbar__logo` 等类名）
- **症状关键词**：logo 图片被放大 / 巨大图片 / 图片失去尺寸约束 / 容器 wrapper 被丢弃
- **问题描述**：开启翻译后，页面左上角的 logo 图片被撑得很大；关闭翻译后页面显示正常。
- **问题原因**：
  - logo 是 `<img class="themedComponent…">`，**自身没有任何尺寸属性**，靠祖先 CSS 定高：
    `.navbar__logo { height: 2rem }` + `.navbar__logo img { height: 100% }`（即高度来自 `.navbar__logo` 这层 wrapper）。
  - DOM：`div.navbar__items`（被认领的翻译块）→ `a.navbar__brand` → **`div.navbar__logo`** → `img`。
  - `lib/extractor.ts` 的 `serializeBlock` 把块内元素分四类，而 `div` 这种"非内联、非块级"的容器落入最后的"其他元素"分支——**只递归子节点、自己不产出任何标记**，于是 `div.navbar__logo` 被**静默拆掉**；重建后这层 wrapper 连同它的 class 一并消失，`.navbar__logo img{height:100%}` 失配，img 回退到自然尺寸 **1772×1799** → 巨大。关闭翻译时不动 DOM，所以正常。
  - 取证：扫样式表确认尺寸规则挂在 `.navbar__logo`；在真实页面手动"拆掉 `div.navbar__logo`"后 logo 从 **32px 暴涨到 522px**，恢复 wrapper 即回 32px。
- **涉及模块**：`lib/extractor.ts` → `serializeBlock`（新增 `hasTranslatableText`、`emitVoid`）
- **解决方案**：序列化时按"子树是否含可翻译文字"分流——
  - **不含可翻译文字的子树**（如只裹图片的 `.navbar__logo`、`a.navbar__brand`）→ 整体按单个 `<xN/>` **深克隆保形**，结构与 class 一律保留（用单个 void 占位也比"空壳 `<gN></gN>`"更稳，模型不会把独立占位删掉）；
  - **含可翻译文字的元素**（内联样式 `<a>/<strong>` 或 `<div>/<section>` 等容器）→ 统一保留属性壳 `<gN>…</gN>` + 递归，**不再静默丢弃容器**。
  - 标记语法（`<gN>`/`<xN>`）未变，故 `markers.ts`/`translator.ts`/`prompt.ts` 无需改动（不动 prompt 以保前缀缓存）。source 字符串自然变化 → 内容寻址缓存对受影响块自动 miss 重译，无需手动清缓存。
- **复现与验证**（回归清单）：
  1. 把 `hermes-agent.nousresearch.com` 加入白名单，打开 `/docs`，等翻译生效（页面出现中文）。
  2. 量 logo 尺寸：`document.querySelector('.navbar__logo img').getBoundingClientRect()` —— **期望 ≈ 32×32**；bug 时会是 ~500+ px。
  3. 结构检查：logo `img` 仍应有 `.navbar__logo` 祖先（`img.closest('.navbar__logo')` 非空）。
  4. 类型检查 `pnpm compile` 通过。
- **经验 / 教训**：
  - **任何承载布局/尺寸的非内联 wrapper（div/figure/section 等），在序列化时都必须保留**，不能当"透明容器"展开——否则依赖该层结构/class 的 CSS 会失配。
  - 靠**祖先选择器**给后代（尤其图片/图标）定尺寸的站点最敏感：Docusaurus、用 Tailwind/CSS-module 工具类的站点都常见 `.wrapper img{height:100%}` 这种写法。
  - 排查"开译后元素样式异常"的通法：对照"关译正常"，定位被翻译块根，**比较重建前后该元素的祖先链有无丢层**。
- **复发记录**：无（2026-06-05 修案例 #2 时同改 `lib/extractor.ts`，已回归校验本案例仍 PASS）

### #2 aria-hidden 阴影副本导致译文丢失、中文竖排溢出

- **发现日期**：2026-06-05
- **状态**：✅ 已修复（2026-06-05）
- **URL**：https://www.firecrawl.dev/
- **站点 / 框架**：www.firecrawl.dev，Tailwind CSS（工具类 `lg:w-max` = `width:max-content`、`px-20`、`relative` 等）
- **症状关键词**：中文竖排 / 文字逐字换行 / 标题溢出 / 宽度塌缩 / aria-hidden 重复副本 / 中英文重叠
- **问题描述**：英文标题水平排列正常；翻译成中文后，该标题文字**竖直排列（逐字一行）并溢出**布局。异常元素是带"阴影副本"的 `<h2>`：
  ```html
  <h2 class="lg:w-max relative mx-auto ... px-20 text-center section-head-title imt-fade-in">
    <div class="overlay -z-[1] p-[inherit] section-head-shadow" aria-hidden="true">We handle the <span class="text-heat-100">hard stuff</span></div>
    We handle the <span class="text-heat-100">hard stuff</span>
  </h2>
  ```
- **问题原因**：该 h2 是常见的"描边/阴影"写法——**一份可见正文 + 一份 `aria-hidden` 的同字绝对定位副本**（叠在背后）。
  - 扩展把**两份重复文字一起送给模型**（source ≈ `<g0>We handle the <g1>hard stuff</g1></g0>We handle the <g2>hard stuff</g2>`）。
  - 模型见到连续重复内容会**"去重"，只回一份、把另一份连标记带文字一起丢掉**；省略成对标记能通过 `validateMarkers`（校验允许不出现全部编号）。
  - 一旦丢的是 **in-flow 那份**、留下的是绝对定位 overlay，则 h2 **没有参与流式布局的内容**：`lg:w-max`（`width:max-content`）塌缩到≈0，overlay 又 `p-[inherit]` 继承 `px-20`（左右各 5rem）使其内容宽也≈0 → 中文在≈0 宽里**逐字换行→竖排+溢出**。英文因单词不可断只会横向溢出、不会竖排，所以"英文正常、中文竖排"。
  - 取证：翻译后该 h2 `childCount=1`（只剩 overlay）、in-flow 文字为空、宽度从 **616px 塌到 40px**。**免 API 受控复现**（克隆该 h2、仅把英文换中文、结构保持完整）渲染为**水平单行**，证明站点 CSS 对中文本身没问题——病根是"结构丢了一份 in-flow 内容"。
- **涉及模块**：`lib/extractor.ts`（`serializeBlock` / `hasTranslatableText` / 新增 `isAriaHidden`）；`entrypoints/content.ts`（新增 `syncAriaHiddenShadows`）
- **解决方案**：两步——
  1. **抽取阶段不把 aria-hidden 内容送翻**：`aria-hidden="true"` 子树按 ARIA 规范属装饰/重复，一律当 `<xN/>` 整体深克隆原样保留（有文字的阴影副本、无文字的图标包装都如此），`hasTranslatableText` 也跳过它。这样重复内容不再进入模型输入，模型无从去重，**in-flow 正文可靠译出、宽度不再塌缩**。
  2. **重建后把阴影副本镜像成译文**（`content.ts` 的 `syncAriaHiddenShadows`）：第 1 步保留的 overlay 仍是英文，会与可见译文重叠成"中英混排"；于是在重建出的 fragment 顶层，找"英文文字 == 原文可见文字（据 source 反推）"的 aria-hidden 副本，用译文节点镜像它。**只镜像确认是重复的副本**，不会误伤"aria-hidden 才是唯一可见文字"等其它用法。
  - 标记语法 / prompt 不变（不动 prompt 保前缀缓存）；source 变化使内容寻址缓存自然 miss 重译。
- **复现与验证**（回归清单）：
  1. `www.firecrawl.dev` 加入白名单，翻译生效。
  2. hero 标题（"我们来处理棘手的事情"）量 `h2.getBoundingClientRect()` → **期望水平单行（w≈555、h≈76）**；bug 时 w≈40、多行。
  3. in-flow 文字非空且为中文；overlay（`.overlay[aria-hidden]`）文字也是**中文（被镜像）、无英文残留**。
  4. 截图肉眼确认：无竖排、无溢出、无中英重叠，`text-heat-100` 橙色 span 样式保留。
  5. `node`（linkedom）结构化测试：#2 全 PASS，且 **#1 仍 PASS**（同模块回归）。
- **经验 / 教训**：
  - **页面里"重复出现的同一段文字"（尤其 aria-hidden 阴影/描边副本、`sr-only` 配对）会诱发模型"去重"丢内容**；要在抽取阶段就把 aria-hidden 装饰/重复内容排除出翻译输入，避免把"防止内容丢失"寄托在模型行为上。
  - **`aria-hidden="true"` = 对辅助技术隐藏的装饰/重复**，按规范不该承载"别处没有的有效信息"→ 不翻译、原样保留是安全且正确的通则（图标包装亦同此理，整体深克隆即可）。
  - **CJK 可在任意两字间断行**：容器内容宽一旦塌到≈0，中文会逐字竖排+溢出，而英文（单词不可断）只横向溢出——**"中文竖排"是"容器宽度塌缩 / in-flow 内容丢失"的强信号**。
  - 与 [#1](#1-docusaurus-logo-图片被放大) 同源：都属 `serializeBlock` 对"块内特殊子元素"的处理（#1 是非内联 wrapper 被拆、#2 是重复/aria-hidden 内容被模型丢）。改这一带逻辑务必同时回归两条。
- **复发记录**：无

### #3 Jest 首页 React #418/#423 hydration 告警（单页、可恢复）

- **发现日期**：2026-06-06
- **状态**：🟡 拆两部分——**(A) #418/#425 外观告警**：观察中、不修（就绪信号时序修复实测无效已回退）；**(B) 晚渲染 SPA ext=0 整页不翻**：✅ **已修复（2026-06-06，`content.ts` 有界沉降补抽）**
- **URL（多站，非单页）**：https://jestjs.io/ 、https://webpack.js.org/concepts/ 、https://stripe.com/ 、https://www.digitalocean.com/community/tutorials/... 、https://hackernoon.com/ 、https://www.mongodb.com/docs/manual/introduction/
- **站点 / 框架**：多个 React-SSR 站（Docusaurus / Next.js / Gatsby / 自建 React）
- **症状关键词**：React #418 / #423 / #425、hydration 文本不匹配、`onRecoverableError`、SPA 内容渲染晚导致 ext=0
- **问题描述**：开启翻译后，部分 React-SSR 站控制台报 `Minified React error #418`（hydration 文本不匹配，可恢复）或 #425/#423 变体。**绝大多数页面仍完整译成中文、布局正常、无致命崩溃**（可恢复，回退客户端渲染后我们的流式回填继续生效）。**例外：MongoDB 文档**首屏正文是客户端渲染（晚于就绪信号），抽取期为空 → **`ext=0` 整页不翻**（功能性失败，非仅外观）。
  - 注：最初只在 Jest 首页观察到，误判为"单页"；全量扫描后确认 **Webpack/Stripe/DigitalOcean/HackerNoon/MongoDB 等多站**都有，是一类问题。
- **问题原因**：
  - `dom-compat.content.ts` 就绪信号用「`load` + `requestIdleCallback`」近似「hydration 已完成」。但这些站把 **hydration / 首屏渲染延迟（或流式）到 load 之后**——首个 idle 仍早于其 hydration。于是就绪信号 → 内容脚本抽取并（命中缓存时毫秒级）回填 → React 随后 hydrate 时见中文 ≠ SSR 英文 → #418/#425；MongoDB 则连正文都还没渲染，抽到 0 块（内容脚本只抽一次）→ ext=0。
  - 取证：`data-imt-ready=1` 却仍 #418；MongoDB `imtReady=1` 但 `extracted=0` 且复查时正文已是英文（渲染晚于抽取）。
- **涉及模块**：`entrypoints/dom-compat.content.ts`（`scheduleReady` 的就绪时机）
- **解决方案（试过并回退，保持现状）**：
  - **尝试**：把就绪信号改为「`load` 后用 MutationObserver 等 DOM 持续静默 500ms（HARD_CAP 5s）再发」，想覆盖延迟 hydration / 晚渲染。
  - **实测无效**（2026-06-06 对照批次）：① 目标站 Jest/Webpack/Stripe/DigitalOcean/HackerNoon 的 hydration 发生在静默窗口**之后**，没被等到，#418 **照旧**；② MongoDB 仍 `ext=0`（渲染更晚）；③ 给每页加 ~500ms 翻译延迟。**对真正目标站零改善 + 加延迟 → 回退**到原 `requestIdleCallback` 版本。
  - **关键机制：#418 与"翻译缓存预热"强相关**（与 CLAUDE.md 已知限制一致："缓存+关思考让译文在 hydration 期间快速注入，会稳定触发"）。同一站缓存**冷**时翻译慢、常在 hydration 后注入 → 干净；缓存**热**时命中毫秒级注入 → 抢在 hydration 前 → #418。**排错陷阱**：firecrawl 首测（冷）干净、复跑（热）#418，曾被我误当"修复回归"——实为缓存预热所致，与改不改就绪信号无关。评估就绪时序改动时**必须固定缓存状态**，否则信号被缓存温度淹没。
  - **结论**：现代站的**延迟/流式/选择性 hydration** + **缓存热快速注入**，无法用内容脚本侧的简单时序启发式可靠规避（React 不暴露干净的 "hydration done" 事件）。#418/#425 是**可恢复**告警、页面仍正确译出；致命的 removeChild 崩溃由 dom-compat 方法补丁**单独**兜住、不依赖此信号时机。为可恢复告警做高风险全局时序改动**不划算**，保持现状。
  - **真要根治 #418 的方向（未来、非平凡）**：需"检测 React 是否完成 hydration"而非"猜时间"——如挂钩 `onRecoverableError` 后重抽重译、或监听特定 root 的 hydration 完成标记。代价与回归面大，暂不做。

- **解决方案 (B)：晚渲染 SPA `ext=0` → ✅ 已修复（有界沉降补抽）**
  - 根因（与 A 同源、但表现是功能性失败）：内容脚本**只抽一次**，而 MongoDB 文档等 SPA 首屏正文是**客户端渐进渲染**（实测 raw 块数 8→52→116，~6-7s 才稳、`readyState` 长时间 `interactive`），抽取时机若早于渲染就抽到 0 → 整页保持英文。已确认内容在**光 DOM、无 shadow/iframe**，是纯时序。
  - 改法（`entrypoints/content.ts`，**不碰 extractor/协议/缓存/prompt**）：初译后跑**有界「沉降-补抽」循环**——最多 5 轮、每轮间隔 1200ms 重抽（`extractBlocks` 的 acceptNode 跳过 `[data-trans-id]` 子树，重跑只返回新块），累积晚到块，某轮 0 新块即判定页面已稳；把累积块在初译 job 结束后用**一个串行 port job** 补译。坑：`extractBlocks` 每次从 b1 重新编号，重抽 id 会撞——故第 ≥1 轮按 `r{round}.b{n}` 重写唯一 id 并改写元素 `data-trans-id`。设计已并入 `front/CLAUDE.md`（数据流 / 已知坑：有界沉降补抽）。
  - 性质：**正常站零影响**（第 1 轮即 0 新块、no-op，实测 MDN/Docusaurus 的 `extracted` 不变、无 `r1.` 前缀块）；**有界自限**（5 轮上限，呼应不上 MutationObserver 的顾虑）；不重复翻译（跳过已认领 + records 按 id 去重）；晚到块一次性补译可接受。
  - 局限：linkedom 的 TreeWalker 不支持 `FILTER_REJECT`，"重抽只返回新块"无法单测，靠端到端验证（真实浏览器 FILTER_REJECT 正常）。`#418` 本身（A）仍在——补抽只解决"抽到 0/不全"，不改初译时机。
- **复现与验证**（回归清单 / 监控项）：
  1. 干净重载（充分预热避开 reload 后首载假象）`https://webpack.js.org/concepts/` 或 `https://jestjs.io/`，控制台会有 `#418`/#425。**可接受判据**：页面最终为中文、无 `removeChild`/`NotFoundError`、无整页 "client-side exception" 崩溃、布局正常。
  2. **MongoDB ext=0 修复回归（B）**：`www.mongodb.com/docs/manual/` 与 `/docs/manual/introduction/` 充分等待（站慢，~16s 才 `readyState=complete`）后应**有译出**（实测各 172/172，修复前为 0）。正常站回归：MDN(`extracted=684` 不变)/react.dev/Docusaurus(#1 logo 32×32) 的 `extracted` 数不变、页面**无 `r1./r2.` 前缀的 data-trans-id**（说明沉降循环对正常站是 no-op）、无重复翻译。
  3. **不要再尝试"MutationObserver 静默改就绪信号"那条路**（A）——已证实无效且被缓存温度淹没（见上）。沉降补抽（B）走的是"初译后有界重抽"，与就绪信号无关，二者别混。
- **经验 / 教训**：
  - **`load`+`requestIdleCallback`/`MutationObserver 静默` 都不是可靠的"hydration 完成"信号**：延迟/流式 hydration 发生在它们之后。
  - **#418 受"缓存温度"主导，会淹没就绪时序的效果**：缓存热时注入快、更易 #418；冷时慢、常干净。评估就绪信号改动**必须固定缓存状态 + 跑含原本干净站的完整对照批次**，否则会把缓存差异误读成"修复有效/回归"（我就把 firecrawl 的缓存热 #418 误当过修复回归）。

### #4 Wikipedia 信息框/窄表格 CJK 标签竖排（2 行）

- **发现日期**：2026-06-06
- **状态**：👀 观察中（**外观问题，经评估暂不修复**，理由见"解决方案"）
- **URL**：https://en.wikipedia.org/wiki/Alan_Turing 、 https://en.wikipedia.org/wiki/Web_browser
- **站点 / 框架**：en.wikipedia.org，MediaWiki（信息框 `.infobox`、统计表）
- **症状关键词**：表格列塌缩、CJK 逐字竖排、`th.infobox-label` 24px、窄单元格、word-break
- **问题描述**：翻译后信息框左侧**标签列**（"教育背景""研究领域""博士论文"等）塌缩到 ~24px 宽、**逐字竖排**（1 字/行）；Web_browser 的浏览器份额表里 td（"UC浏览器""华为浏览器"）同样竖排。正文、值列均正常；非崩溃、不影响阅读主体。
- **问题原因**：**CSS 表格自动布局 + CJK 可任意断行**的固有冲突——
  - 表格列的最小宽度 = 列内"最长不可断单元"。英文标签"Education"最小宽 = 整个词（~65px）；译成"教育背景"后，CJK 字符间可任意断行，最小宽 = **1 个汉字（~24px）**。
  - 于是自动布局把标签列压到 24px、汉字逐字竖排，把省下的宽度让给值列。
  - **取证 / 为何难修**：实测给塌缩的 `th` 加 `white-space:nowrap` 或 `word-break:keep-all`，宽度仅从 24→**58px**（1 字竖排 → 2 字/行），**无法回到单行**——因为该信息框总宽固定 **310px、值列已占 270px**，标签列被硬卡在 ~40px，而中文标签需 ~96px。即**单元格级 CSS 改不动列宽分配**（值列贪占）。注意 `th` 的"高度"= 表格行高（由值单元格撑高），不能用高度判塌缩，**宽度（<~40px）才是可靠信号**。
- **涉及模块**：渲染层；若修，候选在 `entrypoints/content.ts`（回填后对塌缩单元格做修正），不涉及 extractor/prompt/缓存。
- **解决方案**：**当前不改**。权衡：
  - 影响：纯**外观**（可读、不崩溃、不丢内容），且属所有 CJK 翻译工具在 Wikipedia 固定窄信息框上的共性难题。
  - 修复代价/风险：① 单元格级 `nowrap/keep-all` 对固定窄信息框**只能部分改善**（1 字竖排→2 字/行），治标不治本；② 若全局给 `td/th` 上 `word-break:keep-all`，**内容型表格**（单元格含长句）的列会被撑宽 → 整表**横向溢出**，net-negative；③ 真正根治要**重排列宽**（压值列/给标签列 min-width），属侵入式表格布局改动，风险高、收益（外观）低。
  - **候选修复（若将来要做）**：回填后只对**确实塌缩**的短标签单元格（`th,td` 且 `getBoundingClientRect().width < ~40px` 且文本 ≤ ~12 字）施加 `word-break:keep-all`（**用宽度而非高度判定**，避免被行高混淆）；普通窄表格能完全救回，固定窄信息框仍只能 2 字/行。改完必须回归：① 一个内容型长句表格无横向溢出；② #1/#2 仍 PASS。
- **复现与验证**（监控项）：
  1. `en.wikipedia.org` 加白名单，开 `/wiki/Alan_Turing`，翻译生效。
  2. `document.querySelectorAll('th.infobox-label')` 里短标签 `getBoundingClientRect().width` ≈ 24（竖排）。**可接受判据**：正文/值列正常、无内容丢失、无崩溃；标签竖排记为已知外观限制。
  3. 反向监控：普通**内容表格**（非信息框）的窄列若出现 CJK 竖排，且该表不是固定窄宽 → 属"候选修复"能根治的范围，可考虑启用。
- **经验 / 教训**：
  - **CJK + 表格自动布局 = 列易塌缩竖排**：根因是 CJK 最小断行宽度 = 1 字；这是与 [#2](#2-aria-hidden-阴影副本导致译文丢失中文竖排溢出) "中文竖排"**同症状、不同根因**（#2 是 in-flow 内容被模型去重丢失致宽度塌 0；#4 是表格布局把列压窄）。排查竖排先分清是哪类。
  - **判表格单元格塌缩用宽度，不要用高度**：`th/td` 高度 = 行高，会被同行的值单元格撑高，误导。
  - **固定窄宽容器（信息框）里的 CJK 难题不要硬修**：单元格 CSS 动不了列宽分配；为外观做侵入式表格改动得不偿失。
- **复发记录**：无

### #7 列表项带子列表时，父项直接文字漏翻（混合内容硬块放弃认领）

- **发现日期**：2026-06-08
- **状态**：✅ 已修复（2026-06-08）
- **URL**：https://docusaurus.io/docs/installation （Requirements 小节那条列表项）
- **站点 / 框架**：docusaurus.io，**Docusaurus**（React-SSR，预渲染正文）
- **症状关键词**：列表项带子列表 / `li` 嵌 `ul` / 混合内容 / 父项直接文字漏翻 / 紧凑列表项 / 父块放弃认领 / 直接文本节点不译
- **问题描述**：开启翻译后，Requirements 列表那条 `<li>`（"Node.js version 20.0 or above … nvm to manage … single machine."）**整段保持英文不译**；而它**嵌套的子列表项**（"When installing Node.js, check all checkboxes…"）却被正常译出。关闭翻译时一切正常。
- **问题原因**（根因，附取证）：
  - 该 `<li>` 是**混合内容**：既有大段**直接文字 / 内联内容**（裸文本节点 + `<a>Node.js</a>`、行内 `<code>node -v</code>`、`<a>nvm</a>`），又**嵌套了子列表 `<ul><li>…</li></ul>`**。
  - `lib/extractor.ts` `extractBlocks` 找候选块根时，硬块（`LI`）的认领条件是 `claim = !hasDescendantHardBlock(el)`。该 li 含后代硬块（子列表的 `<li>`）→ `hasDescendantHardBlock=true` → **父 li 放弃认领自己**，让 TreeWalker 深入、由子 `<li>` 各自认领翻译。但父 li 自己那段**直接文字游离在子 `<ul>` 之外、不属于任何深层硬块** → **无人认领 → 漏翻**。
  - 取证：linkedom 解析真实页面，父 li `querySelector('p,li,h1..h6,td,th,blockquote,dt,dd,figcaption,summary,caption')` 命中子列表 li（后代硬块数=1）；用与 `hasDescendantHardBlock` **逐字等价**的判定，父 li `claim=false`、子 li `claim=true`，父 li 直接文本即那段 Node.js 说明。**爆炸半径扫描**：整页（含导航/TOC）仅 **3 个**这种"含直接文字 + 含后代硬块"的硬块（本案 li、Project structure rundown 的 `/src/` li、右侧 TOC 一个带子项的目录项），文本 ≤405 字、内含硬块 1–2 个、浅嵌套 → 无巨型块风险。
  - **与松散列表区分（关键）**：Markdown **松散列表** `<li><p>…</p></li>` 的文字**都在 `<p>` 里、`<li>` 无直接文字**（`hasDirectText=false`），原逻辑让深层 `<p>` 认领、**不漏**；本案是**紧凑列表项 + 子列表**，直接文字**裸挂在 `<li>` 上**才暴露盲区。
- **涉及模块**：`lib/extractor.ts` → `extractBlocks` 的硬块 `claim` 判定。
- **解决方案**：硬块认领条件改为 `claim = !hasDescendantHardBlock(el) || hasDirectText(el)`——含后代硬块但**自己还有直接文字**时，由**自己认领整体**。
  - `serializeBlock` 把嵌套子列表当普通容器（`hasTranslatableText=true` → `<gN>` 保壳递归），子项文字**并入同一块**翻译；TreeWalker `acceptNode` 的 `closest('[data-trans-id]')` 会跳过已认领子树，**子 li 不再重复认领**。实测父块 source 形如 `<g0>Node.js</g0> version 20.0…<x1/>… <g2>nvm</g2> … <g3><g4>When installing…</g4></g3>`（成对标记合法、`<code>` 走 `<x1/>` 原样保留）。
  - **软块不放开**同样的「有直接文字就认领整体」——`div/main/section` 等可能是整页级容器，认领整体会把大量后代硬块吞成巨型块；混合内容只在**硬块**（粒度天然可控）上处理。
  - **不动 prompt / 标记协议（`<gN>`/`<xN>`）/ 缓存键**；受影响块 source 自然变化 → 内容寻址缓存自动 miss 重译，无需清缓存。
- **复现与验证**（回归清单）：
  1. `docusaurus.io` 加白名单，开 `/docs/installation`，待译出。定位 Requirements 的 li（含 `node -v` 的那条 `ul > li`）。**期望**：该 li 有 `data-trans-id`；其**直接文本节点含中文**（"版本 20.0 或以上（可通过运行 … 检查）…"）；**子 li 无独立 `data-trans-id`**（被父整体认领）；行内 `node -v` **保英文**（铁律 #6）、`nvm` 链接保留。实测 `b74`：父 li 全中文、子列表"安装 Node.js 时…"并入同块、`subLiHasOwnTransId=false`、整页 67 块 33 中文无崩溃。**bug 时**：该 li 无 `data-trans-id`、直接文字保持英文。
  2. **离线全链路**（`node` + linkedom；esbuild plugin 在 bundle 时给 `extractor.ts` 注入内部导出、**不改源码**，跑真实 `serializeBlock`/`validateMarkers`/`rebuild`）：混合内容 li 认领后 source 标记校验通过、rebuild 还原 **2 个 `<a>` + `<code>` + 嵌套 `<ul><li>`**、子列表文字在块内；**对照** 松散列表 `li→p` 仍 2 个 `<p>` 各自认领（不退化大块）、普通列表不变。
  3. **同模块（`extractor.ts`）回归 [#1](#1-docusaurus-logo-图片被放大) / [#2](#2-aria-hidden-阴影副本导致译文丢失中文竖排溢出)**：hermes-agent logo **32×32 且仍在 `.navbar__logo` wrapper 内**；firecrawl hero h2 **水平 555×76（ratio 7.3、不竖排）**、in-flow 中文、aria-hidden overlay 镜像中文——实测全 PASS（本改动只动硬块认领，不碰软块 wrapper / aria-hidden 路径）。
  4. `pnpm compile` + `pnpm build` 通过。
- **经验 / 教训**：
  - **「同时含直接文字 + 嵌套硬块」是抽取盲区**：原 `claim=!hasDescendantHardBlock` 只设想"要么自己翻、要么全交深层"，漏了"自己有一部分文字、另一部分是深层块"的混合情形。补 `|| hasDirectText` 让这种块认领整体（连子块一起翻，上下文更完整）。与 [#1](#1-docusaurus-logo-图片被放大) 同源——都是 `extractBlocks`/`serializeBlock` 对「块内特殊结构」处理不周（#1 丢 wrapper、本条漏直接文字）。
  - **`hasDirectText` 精确区分紧凑列表项 vs 松散列表**：只有直接文字**裸挂在硬块上**才认领整体；松散列表 `li→p` 的 li 无直接文字，不受影响——这是不破坏 Markdown 松散列表的关键，改这一带务必回归松散列表用例。
  - **linkedom 跑不了 `extractBlocks`**：其 `createTreeWalker` **根本不回调 `acceptNode`**（比经验库 [#3](#3-jest-首页-react-418423-hydration-告警单页可恢复) 记的"不支持 FILTER_REJECT"更彻底，`nextNode` 直接返空）。抽取的**认领遍历只能靠真实浏览器**；但 `serializeBlock`/`rebuild`/`validateMarkers` 是纯递归/纯函数，可离线测——用 esbuild `onLoad` plugin 给入口注入内部导出即可，**不必污染源文件**。
  - **调试 Chrome「多扩展同名 + MV3 SW idle」陷阱**（排错弯路，记之）：本机同时装了别人的同名商业「沉浸式翻译」(DISABLED) 和我们的 DeepSeek 版，且 MV3 **SW 空闲终止后不在 `/json`**——若按 `background.js` 猜 SW，会连错扩展（我一度把白名单/reload 全打到「iCloud 密码」扩展上，导致页面 content script 注入了却 `total=0` 不译）。**务必用 `chrome://extensions` 的 `developerPrivate.getExtensionsInfo` 按 `manifest.name` 锁定 id**，并核对其加载 `path` = `output/chrome-mv3`；SW idle 时改用扩展自己的 `options.html` 写 storage。
